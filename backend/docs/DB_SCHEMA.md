# Database Schema

## Overview

This document defines the NoSQL schema for journalist-created articles stored in **Firebase Firestore**, with thumbnail images stored in **Firebase Cloud Storage**.

The schema is intentionally flat (no subcollections) because the data access pattern is simple: list all articles, create one article. There are no relational joins, no user-scoped queries, and no pagination beyond Firestore's built-in cursor support. A flat collection is the simplest, most performant choice for this use case.

---

## Firestore Collections

### `articles`

Top-level collection storing journalist-created articles.

| Field | Type | Required | Description | Constraints |
|-------|------|----------|-------------|-------------|
| `title` | `string` | Yes | Article headline | 1-200 characters |
| `description` | `string` | Yes | Short summary / subtitle | 1-500 characters |
| `content` | `string` | Yes | Full article body text | 1-10,000 characters |
| `author` | `string` | Yes | Author display name | 1-100 characters |
| `thumbnailURL` | `string` | Yes | Download URL of the thumbnail image in Cloud Storage | Must be a valid HTTPS URL pointing to `media/articles/` path |
| `createdAt` | `timestamp` | Yes | Server-side creation timestamp | Set on write, immutable after creation |

**Document ID:** Auto-generated by Firestore (`doc()` with no argument).

#### Example Document

```json
{
  "title": "Breaking: Flutter 4.0 Released",
  "description": "Google announces the next major version of Flutter with native desktop support.",
  "content": "In a keynote at Google I/O 2026, the Flutter team announced...",
  "author": "Diego Cardenas",
  "thumbnailURL": "https://firebasestorage.googleapis.com/v0/b/symmetry-news-app-dac91.firebasestorage.app/o/media%2Farticles%2F1711367890123_flutter4.jpg?alt=media&token=abc123",
  "createdAt": "2026-03-25T14:30:00Z"
}
```

#### Indexes

No custom composite indexes are required. The only query pattern is listing all articles ordered by `createdAt` descending, which uses the automatic single-field index on `createdAt`.

## Cloud Storage Structure

```
media/
  articles/
    {timestamp}_{sanitized_filename}.{ext}
```

### Path Convention

- **Base path:** `media/articles/`
- **Filename format:** `{Unix milliseconds timestamp}_{original filename}`
- **Allowed extensions:** `.jpg`, `.jpeg`, `.png`, `.webp`
- **Max file size:** 5 MB (enforced by Storage rules)

### Example Paths

```
media/articles/1711367890123_flutter4.jpg
media/articles/1711368000000_breaking_news.png
```

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Flat collection** (no subcollections) | Single access pattern (list + create); no user-scoped queries needed. Simplest, fastest reads. |
| **No auth fields** (`userId`, etc.) | Assignment doesn't require authentication. v1 is open-write. Security rules enforce schema validation only. |
| **Server timestamp for `createdAt`** | Prevents client clock skew. Immutable after creation (enforced by rules). |
| **`thumbnailURL` as full download URL** | Client stores the complete download URL returned by `getDownloadURL()`. Avoids extra Storage lookups at read time. |
| **Timestamp prefix on filenames** | Guarantees uniqueness without UUID dependency. Human-readable for debugging. |
| **5 MB image limit** | Balances quality with mobile upload speed. Client-side compression targets <500KB before upload; 5MB is the server-side safety net. |
| **No `id` field in document** | Firestore document ID serves as the unique identifier. Storing it redundantly inside the document is an anti-pattern. |

---

## Schema Enforcement

- **Firestore rules** in `backend/firestore.rules` validate field presence, types, string lengths, and timestamp immutability
- **Storage rules** in `backend/storage.rules` validate file path, content type (image/*), and max size (5 MB)

See those files for the full rule implementations.
