# Database Schema

## Overview

This document defines the NoSQL schema for journalist-created articles stored in **Firebase Firestore**, with thumbnail images stored in **Firebase Cloud Storage**.

The schema is intentionally flat (no subcollections) because the data access pattern is simple: list all articles, create one article, update own articles. A flat collection is the simplest, most performant choice for this use case.

---

## Firestore Collections

### `articles`

Top-level collection storing journalist-created articles.

| Field | Type | Required | Mutable | Description | Constraints |
|-------|------|----------|---------|-------------|-------------|
| `title` | `string` | Yes | Yes | Article headline | 1-200 characters |
| `description` | `string` | Yes | Yes | Short summary / subtitle | 1-500 characters |
| `content` | `string` | Yes | Yes | Full article body text | 1-10,000 characters |
| `author` | `string` | Yes | Yes | Author display name | 1-100 characters |
| `thumbnailURL` | `string` | Yes | Yes | Download URL of the thumbnail image in Cloud Storage | Must be a valid HTTPS URL pointing to `media/articles/` path |
| `ownerUid` | `string` | Yes | **No** | Firebase Auth UID of the article creator | Set on create, immutable. Used for ownership enforcement in security rules and "My Articles" queries. |
| `createdAt` | `timestamp` | Yes | **No** | Server-side creation timestamp | Set on write via `FieldValue.serverTimestamp()`, immutable after creation |
| `category` | `string` | No | Yes | Article category (e.g., "technology", "sports") | 1-50 characters if present. Optional — omitted when user does not select a category. |

**Document ID:** Auto-generated by Firestore (`doc()` with no argument).

#### Example Document

```json
{
  "title": "Breaking: Flutter 4.0 Released",
  "description": "Google announces the next major version of Flutter with native desktop support.",
  "content": "In a keynote at Google I/O 2026, the Flutter team announced...",
  "author": "Diego Cardenas",
  "ownerUid": "abc123def456",
  "thumbnailURL": "https://firebasestorage.googleapis.com/v0/b/symmetry-news-app-dac91.firebasestorage.app/o/media%2Farticles%2F1711367890123_flutter4.jpg?alt=media&token=abc123",
  "category": "technology",
  "createdAt": "2026-03-25T14:30:00Z"
}
```

#### Indexes

| Fields | Order | Purpose |
|--------|-------|---------|
| `ownerUid`, `createdAt` | `ownerUid` ASC, `createdAt` DESC | "My Articles" screen — list articles by the authenticated user, newest first |

The single-field automatic index on `createdAt` covers the home feed (all articles, newest first).

## Cloud Storage Structure

```
media/
  articles/
    {timestamp}_{sanitized_filename}.{ext}
```

### Path Convention

- **Base path:** `media/articles/`
- **Filename format:** `{Unix milliseconds timestamp}_{original filename}`
- **Allowed extensions:** `.jpg`, `.jpeg`, `.png`, `.webp`
- **Max file size:** 5 MB (enforced by Storage rules)

### Example Paths

```
media/articles/1711367890123_flutter4.jpg
media/articles/1711368000000_breaking_news.png
```

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Flat collection** (no subcollections) | Single access pattern (list + create + update own); no complex relational queries needed. Simplest, fastest reads. |
| **`ownerUid` for ownership** | Stable, immutable identifier tied to Firebase Auth. Unlike `author` (display name), UIDs cannot collide or be impersonated. Enables secure "My Articles" queries and owner-only update rules. |
| **`category` as optional field** | Not all articles require categorization. `hasOnly` in Firestore rules permits documents with or without this field. |
| **Server timestamp for `createdAt`** | Prevents client clock skew. Immutable after creation (enforced by rules via `newData.createdAt == existingData.createdAt`). |
| **`thumbnailURL` as full download URL** | Client stores the complete download URL returned by `getDownloadURL()`. Avoids extra Storage lookups at read time. |
| **Timestamp prefix on filenames** | Guarantees uniqueness without UUID dependency. Human-readable for debugging. |
| **5 MB image limit** | Balances quality with mobile upload speed. Client-side compression targets <500KB before upload; 5MB is the server-side safety net. |
| **No `id` field in document** | Firestore document ID serves as the unique identifier. Storing it redundantly inside the document is an anti-pattern. |
| **Owner-based updates, no deletes** | Allows authors to fix typos and update content while preventing data loss. `ownerUid` and `createdAt` are immutable on update. |

## Schema Enforcement

- **Firestore rules** in `backend/firestore.rules` validate:
  - Field presence and types (required + optional)
  - String length constraints
  - `ownerUid == request.auth.uid` on create
  - `resource.data.ownerUid == request.auth.uid` on update
  - `ownerUid` and `createdAt` immutability on update
  - Server timestamp enforcement
- **Storage rules** in `backend/storage.rules` validate file path, content type (image/*), and max size (5 MB)

See those files for the full rule implementations.
