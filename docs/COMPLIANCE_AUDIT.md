## Objective

This document verifies that the codebase complies with three governing documents provided by Symmetry Club OU:

1. **Assignment Requirements** ([ASSIGNMENT_REQUIREMENTS.md](./ASSIGNMENT_REQUIREMENTS.md)) — The deliverables, architecture constraints, and evaluation criteria for the job application assignment.
2. **Architecture Violations** ([ARCHITECTURE_VIOLATIONS.md](./ARCHITECTURE_VIOLATIONS.md)) — 50 specific violations to avoid across the data, domain, and presentation layers.
3. **Coding Guidelines** ([CODING_GUIDELINES.md](./CODING_GUIDELINES.md)) — 6 coding standards enforced on a per-pull-request basis.

Each requirement is evaluated as **PASS**, **FLAG** (non-blocking concern), or **FAIL** (blocks submission).

## 2. Architecture Violations Compliance

### 2.1 Data Layer (AV 1.x)

| Rule | Description | Status | Notes |
|------|-------------|--------|-------|
| 1.1.1 | Never import from presentation layer | **PASS** | Verified: no presentation imports in any data layer file. |
| 1.1.2 | Do not import use_cases | **PASS** | Verified: no use_case imports in data layer. |
| 1.2.1 | Throw exceptions from data_sources | **PASS** | Data sources throw on failure; repository catches and wraps in `AppException`. |
| 1.2.2 | Data sources convert API data to models or trigger external changes only | **PASS** | `FirestoreArticleDataSourceImpl` creates Firestore docs and returns models. `StorageArticleDataSourceImpl` uploads files and returns URLs. `NewsApiService` fetches and returns models. |
| 1.2.3 | Data sources are the ONLY place connecting to external devices | **PASS** | All Firebase and API calls isolated in data_sources. |
| 1.2.4 | Data sources are the ONLY place to import providers (Firebase, Dio, etc.) | **FLAG** | `injection_container.dart` imports `package:dio`, `package:cloud_firestore`, and `package:firebase_storage` for DI wiring. This is the DI container (not a clean layer), so arguably exempt — but a strict reading could flag it. See [Flagged Item #10](#flagged-items). |
| 1.3.1 | Models ALWAYS extend an entity | **PASS** | `FirebaseArticleModel extends FirebaseArticleEntity`. `ArticleModel extends ArticleEntity`. |
| 1.3.2 | Models contain `toEntity()` function | **PASS** | Both models implement `toEntity()`. |
| 1.3.3 | Models contain `fromRawData` factory | **FLAG** | `FirebaseArticleModel` uses `fromRawData` (compliant). `ArticleModel` uses `fromJson` because Retrofit code generation requires this name. See [Flagged Item #2](#flagged-items). |
| 1.4.1 | Repository implementations named `{Interface}Impl` | **PASS** | `CreateArticleRepositoryImpl`, `ArticleRepositoryImpl`. |
| 1.4.2 | Only repository implementations of domain abstract classes | **PASS** | Both repo impls implement their corresponding domain interfaces. |
| 1.4.3 | ALWAYS return `DataState<Type>` when requesting data from an API | **FLAG** | `CreateArticleRepository` returns `DataState` for both methods (compliant). `ArticleRepository.getSavedArticles()` returns `List<ArticleEntity>` without `DataState` wrapping — this is a local DB call, not an API call, so arguably exempt. See [Flagged Item #9](#flagged-items). |
| 1.4.4 | Only repos import data_sources | **PASS** | Verified: only repository files import data_source files. |

## 3. Coding Guidelines Compliance

| Rule | Description | Status | Notes |
|------|-------------|--------|-------|
| CG1 | **Boy Scout Rule** — Leave code cleaner than you found it | **PASS** | 10 existing issues identified and fixed. Documented in `REFACTOR_REPORT.md`. New feature code is cleaner than existing patterns. |
| CG2 | **Meaningful names** — Intention-revealing, pronounceable, convention-following | **PASS** | All classes, methods, variables follow descriptive naming. Class names are nouns (`FirebaseArticleEntity`, `CreateArticleCubit`). Function names are verbs (`uploadImage`, `submitArticle`, `createArticle`). |
| CG2.1 | Follow set conventions for typical files | **FLAG** | `ArticleModel.fromJson` deviates from the `fromRawData` convention (Retrofit requirement). `RemoteArticlesEvent` does not extend `Equatable` while `LocalArticlesEvent` does — inconsistent. See Flagged Items [#2](#flagged-items) and [#8](#flagged-items). |
| CG3 | **Small functions** — SRP, low nesting, descriptive names, few arguments | **PASS** | Pages decomposed into `_buildAppBar`, `_buildBody`, `_buildFormFields`, `_buildImageSection`, `_buildSubmitButton`, etc. Max nesting depth ≤ 2. |
| CG3.5 | Keep number of arguments low | **PASS** | Use case `call()` takes 1 param. Cubit methods take 0–5 named params. Widget constructors use named params (acceptable per Flutter convention). |
| CG3.6 | Command Query Separation | **PASS** | Functions either mutate state (`uploadImage`, `submitArticle`) or return data (`toEntity`, `fromRawData`), not both. |
| CG4 | **TDD** — Test all functionality | **FLAG** | 55 tests across 9 test files covering all `create_article` layers. However, existing `daily_news` use cases, BLoCs, and repository have zero tests. CG1 states: "If we update functionality which was not tested, we create the tests ourselves." See [Flagged Item #7](#flagged-items). |
| CG4.2 | Integration tests for user journeys | **FLAG** | No integration tests exist. Widget tests cover individual components but not end-to-end flows. |
| CG5 | **Small classes** — Single responsibility, few instance variables | **PASS** | All classes have a single purpose. `CreateArticleCubit` has 2 instance variables. Entity/model classes match their schema exactly. |
| CG6 | **Abstract classes** for isolation | **PASS** | `UseCase<Type, Params>`, `CreateArticleRepository`, `ArticleRepository`, `FirestoreArticleDataSource`, `StorageArticleDataSource` — all abstract interfaces enabling testability and layer isolation. |
